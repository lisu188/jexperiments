# Kotlin Class and Property Delegation Bytecode

Kotlin delegation compresses forwarding and property behavior in source, but the class file still contains delegate fields, forwarding methods, `KProperty` arrays, and calls to `getValue` and `setValue`. This experiment shows class delegation and property delegation together.

## Why This Matters

Java developers reading Kotlin-generated bytecode need to know where forwarding methods come from. Kotlin developers need to know that delegated properties are not magic fields. Library authors need to watch hidden allocations and generated members because delegation choices become class layout.

The demo is safe. It constructs in-memory delegates, invokes methods, and prints deterministic messages.

## Execution Path

Class delegation forwards `Reporter` calls:

```kotlin
interface Reporter {
    fun report(message: String): String
}

class ReportingService(delegate: Reporter) : Reporter by delegate {
    fun ownReport(): String = report("own")
}
```

Property delegation sends reads and writes to operator functions:

```kotlin
class AuditPropertyDelegate(private var value: String) {
    operator fun getValue(thisRef: Any?, property: KProperty<*>): String {
        println("getValue(${property.name})")
        return value
    }

    operator fun setValue(thisRef: Any?, property: KProperty<*>, newValue: String) {
        println("setValue(${property.name},$newValue)")
        value = newValue
    }
}
```

The delegated class combines a custom delegate and `lazy`:

```kotlin
class DelegatedProperties {
    var status: String by AuditPropertyDelegate("new")
    val summary: String by lazy { "summary-$status" }
}
```

## Bytecode Landmarks

`ReportingService` contains:

```text
private final Reporter $$delegate_0
public String report(String)
```

The generated `report` method loads `$$delegate_0` and invokes `Reporter.report` with `invokeinterface`. `ownReport` calls the generated `report`, so even code inside the class observes the forwarding method.

`DelegatedProperties` contains:

```text
static final KProperty[] $$delegatedProperties
private final AuditPropertyDelegate status$delegate
private final Lazy summary$delegate
```

`getStatus` calls `AuditPropertyDelegate.getValue(this, $$delegatedProperties[0])`. `setStatus` calls `setValue`. `getSummary` calls `Lazy.getValue()`.

## Interop Caveats

Java sees ordinary fields and methods, not Kotlin's delegation syntax. The synthetic `$$delegate_0` field is private but visible to bytecode tools. Property delegate fields are implementation details; changing the delegate type changes class layout.

Delegated properties can hide allocations. `lazy` creates a `Lazy` object, and custom delegates usually create delegate instances. That may be fine, but it should be intentional in library types.
