# Kotlin Default Arguments at the JVM Boundary

Kotlin default parameters look source-level, but the JVM has no default-argument calling convention. This experiment shows how Kotlin 2.3.0 lowers default constructor and method arguments into synthetic `$default` methods, bit masks, overloads from `@JvmOverloads`, and `DefaultConstructorMarker` constructor shapes that Java developers will see during bytecode inspection.

## Why This Matters

Java developers need to know why a Kotlin API may expose more methods than the source declares. Kotlin developers need to know which defaults are binary ABI and which overloads are for Java source convenience. Library authors need to treat `$default` descriptors and overload sets as compatibility surface, because callers compiled against them can keep referencing those exact methods.

The demo is safe to run on any normal JDK. It performs no network, file-system mutation, hardware access, or destructive operation; it only prints deterministic strings.

## Execution Path

The constructor has two defaulted parameters and is annotated with `@JvmOverloads`:

```kotlin
class KotlinGreeter @JvmOverloads constructor(
    private val prefix: String = "hello",
    private val punctuation: String = "!"
)
```

The method also has two defaulted parameters:

```kotlin
@JvmOverloads
fun greet(name: String = "bytecode", times: Int = 1): String =
    (1..times).joinToString(separator = "|") { "$prefix $name$punctuation" }
```

The Java caller deliberately uses the generated overloads, not Kotlin's named-argument syntax:

```java
KotlinGreeter greeter = new KotlinGreeter("java");
return KotlinDefaults.label("caller") + "|" + greeter.greet("reader");
```

The output proves that Kotlin defaults and Java overloads reach the same implementation:

```text
kotlin-default-constructor=hello bytecode!
kotlin-default-method=hi bytecode!|hi bytecode!
kotlin-static-default=module-1-demo
java-overload-call=caller-1-ok|java reader!
```

## Bytecode Landmarks

`KotlinGreeter` contains a synthetic constructor with this descriptor:

```text
(Ljava/lang/String;Ljava/lang/String;ILkotlin/jvm/internal/DefaultConstructorMarker;)V
```

The extra `int` is the mask. Bit `1` means use the default for `prefix`; bit `2` means use the default for `punctuation`. The trailing `DefaultConstructorMarker` exists so the synthetic constructor does not collide with a user-declared constructor.

The method default bridge is visible as:

```text
public static java.lang.String greet$default(KotlinGreeter, String, int, int, Object)
flags: ACC_PUBLIC, ACC_STATIC, ACC_SYNTHETIC
```

`@JvmOverloads` adds real Java-callable overloads:

```text
greet(String, int)
greet(String)
greet()
```

`KotlinDefaults.label` shows the same pattern on an `object` method. Because the function is also `@JvmStatic`, Java sees static overloads on `KotlinDefaults`, while Kotlin still has the object singleton model.

## Interop Caveats

Java source cannot call Kotlin defaults directly. It can call only the overloads generated by `@JvmOverloads`, or it can use reflection to reach synthetic `$default` methods. The latter is fragile: mask positions are tied to parameter order, and changing parameter order or removing a default is a binary compatibility event.

Kotlin named arguments are not represented in the JVM descriptor. Java callers see only positional overloads. Nullability is encoded through Kotlin metadata and runtime checks, not Java's method descriptor.
